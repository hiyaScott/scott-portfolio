## 1. 系统概述

### 1.1 产品定位
认知负载监控系统（Cognitive Load Monitor）是一个实时展示 AI Agent 运行状态的可视化仪表盘。通过分析会话活跃度、消息等待时间、Token 消耗量等指标，为用户提供一个直观的"健康状态"视图。

### 1.2 核心价值
- **透明度**: 让用户了解 Agent 当前的工作负载
- **预期管理**: 通过预计响应时间帮助用户合理安排任务
- **历史追踪**: 记录负载变化趋势，支持性能分析

## 2. 功能详细说明

### 2.1 核心功能模块

#### 2.1.1 实时状态仪表盘 (Radio Tuner)

| 功能项 | 说明 | 数据来源 |
|--------|------|----------|
| 认知评分 | 0-100% 连续值，综合反映当前负载 | 混合算法计算 |
| 状态指示灯 | 四色分区：空闲(绿)/轻载(蓝)/中等(黄)/高载(红) | 评分映射 |
| 动态指针 | 收音机式指针，实时指向当前负载位置 | CSS动画 |
| 状态文本 | 文字描述当前状态和建议 | 评分区间映射 |

**评分算法 (Mixed Score Algorithm)**:
```
最终评分 = 基础评分 + max(等待评分, Token评分) + 处理加成

基础评分: 50 (恒定)
等待评分: 基于最长等待消息的时间 (0-95分)
  - 0-10s: 20-30%
  - 10-30s: 30-55%
  - 30-60s: 55-80%
  - 60s+: 80-95%
  
Token评分: 只计算处理中任务的Token量 (0-20分)
  - 10k: ~10%
  - 50k: ~30%
  - 100k: ~50%
  - 200k+: 75-80%

处理加成: 每个处理中任务 +3% (最多+15%)
```

#### 2.1.2 状态栏 (Status Bar)

| 指标 | 说明 | 更新频率 |
|------|------|----------|
| 连接状态 | 实时/离线/高负载 | 每次刷新 |
| 频道 | 当前消息来源 (feishu/telegram等) | 静态配置 |
| 模型 | 当前使用的AI模型 (k2p5等) | 静态配置 |
| Agent状态 | active/busy/error | 每次刷新 |
| CPU使用率 | 服务器CPU占用百分比 | 每次刷新 |
| 内存占用 | 服务器内存占用百分比 | 每次刷新 |

#### 2.1.3 指标卡片 (Metrics Grid)

| 指标 | 说明 | 单位 |
|------|------|------|
| 活跃会话 | 最近10分钟内有活动的会话数 | 个 |
| 待处理 | 用户已发消息但未开始处理的数量 | 个 |
| 处理中 | 当前正在执行的任务数 | 个 |
| 最长等待 | 最久一条消息等待回复的时间 | 秒/分钟 |
| 预计响应 | 基于当前负载估算的响应时间 | 秒/分钟 |

**预计响应时间算法**:
```python
estimated_response = processing_count * 30  # 基础时间
                     + min(60, tokens/50000 * 30)  # Token加成
                     + pending_count * 15  # 排队时间
```

## 4. 安全风险评估

### 4.1 风险等级矩阵

| 风险项 | 严重程度 | 可能性 | 风险等级 | 状态 |
|--------|----------|--------|----------|------|
| Redis Token 泄露 | 高 | 已发生 | 🔴 严重 | 接受风险 |
| CORS 跨域攻击 | 中 | 已发生 | 🟡 中 | 接受风险 |
| 敏感信息泄露 | 低 | 低 | 🟢 低 | 可控 |
| 服务可用性 | 中 | 中 | 🟡 中 | 监控中 |

### 4.2 详细风险说明

#### 🔴 严重: Redis Token 泄露

**风险描述**:
- Upstash Redis 的访问 Token 硬编码在前端 JavaScript 中
- 任何访问页面的用户都可以在 DevTools 中看到 Token

**影响范围**:
```javascript
// 暴露的凭证
const config = {
    url: 'https://singular-snake-71209.upstash.io',
    token: 'gQAAAAAAARYpAAIncDE2NmRhOGU0OWFhZWM0N2I4OGZlMGZkNGM5NjdjMTI5NnAxNzEyMDk'
};
```

**潜在威胁**:
1. 恶意用户可直接访问 Redis 读取/写入数据
2. 数据被篡改可能导致监控显示虚假信息
3. 如果 Redis 中存储敏感信息，可能被窃取

**缓解措施**:
- Upstash Token 仅具有读取权限（理论上）
- Redis Key 设计为单一用途，不影响其他系统
- 数据为临时状态数据，不含敏感信息

**修复方案（未实施）**:
- 方案A: 服务器端 API 代理（需要 Nginx 配置）
- 方案B: 使用 GitHub Actions 或 Vercel Edge Function 做代理
- 方案C: 迁移到支持 CORS 的数据源

#### 🟡 中: CORS 跨域问题

**风险描述**:
- 浏览器安全策略阻止直接访问第三方 Redis 服务
- 导致页面在某些环境下显示"连接失败"

**发生场景**:
- 文件拆分时 JS 单独加载，CORS 预检失败
- 用户浏览器启用严格安全策略

**当前状态**:
- 已回滚到单文件内联模式，CORS 问题暂时解决
- 单文件模式下 fetch 请求可正常执行

#### 🟢 低: 敏感信息泄露

**风险评估**:
- 监控数据仅包含会话活跃度、Token量等元数据
- 不包含用户消息内容、个人信息等敏感数据
- 风险可控

#### 🟡 中: 服务可用性

**风险描述**:
- 依赖 Upstash Redis 服务可用性
- 如果 Upstash 服务中断，监控页面将显示"连接失败"
- 监控服务进程可能崩溃或停止

**监控措施**:
- 页面显示"连接失败"提示用户
- 建议添加服务健康检查告警

### 4.3 安全建议

#### 短期（1周内）
1. **降级 Token 权限**: 确认 Upstash Token 仅有读取权限，移除写入权限
2. **数据隔离**: 确保 Redis 中只存储非敏感的状态数据
3. **监控告警**: 添加监控服务进程保活机制

#### 中期（1个月内）
1. **API 代理部署**: 部署 Nginx 反向代理，隐藏 Token
2. **访问控制**: 添加简单的访问频率限制
3. **日志审计**: 记录异常访问行为

#### 长期（3个月内）
1. **认证机制**: 添加简单的 Token 认证或 IP 白名单
2. **数据加密**: 敏感配置使用环境变量管理
3. **架构升级**: 考虑迁移到 Vercel/Cloudflare Workers 做 Edge 计算
